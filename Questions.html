Based on technologies from CV: React, TypeScript, Redux Toolkit, WebSockets, Playwright, Webpack, and more...

React.js

Q1: Explain the difference between class components and functional components with hooks.

Class Components: Use ES6 classes, have lifecycle methods (componentDidMount, componentDidUpdate, etc.), use 'this' keyword, and require more boilerplate code.

Functional Components with Hooks: Simpler syntax, use hooks (useState, useEffect, etc.) for state and lifecycle, easier to test and understand, and are the recommended approach in modern React (React 16.8+).

Hooks allow functional components to have state and side effects, making them equivalent to class components but with cleaner code.

Q2: What are React Hooks? Name the most commonly used hooks and their purposes.

React Hooks are functions that let you use state and other React features in functional components.

  • useState: Manages component state
  • useEffect: Handles side effects (API calls, subscriptions, DOM manipulation)
  • useContext: Accesses React context
  • useReducer: Alternative to useState for complex state logic
  • useMemo: Memoizes expensive calculations
  • useCallback: Memoizes functions to prevent unnecessary re-renders
  • useRef: Creates mutable references that persist across renders
Q3: Explain the useEffect hook and its dependency array.

useEffect runs after render and allows you to perform side effects. The dependency array controls when the effect runs:

  • No array: Runs after every render
  • Empty array []: Runs only once after initial render (like componentDidMount)
  • With dependencies [a, b]: Runs when a or b changes
useEffect(() => { // Side effect code return () => { // Cleanup function (like componentWillUnmount) }; }, [dependency1, dependency2]);
Q4: How do you optimize React performance?

React performance optimization involves preventing unnecessary re-renders and expensive computations. Here are the key techniques:

1. React.memo - Component Memoization

React.memo is a higher-order component that memoizes the result of a component. It only re-renders if props have changed (shallow comparison).

const ExpensiveComponent = React.memo(({ name, count }) => { console.log('Rendering ExpensiveComponent'); return ( <div> <h2>{name}</h2> <p>Count: {count}</p> </div> ); }); const ExpensiveComponentWithCustomComparison = React.memo( ({ name, count }) => { return <div>{name}: {count}</div>; }, (prevProps, nextProps) => { return prevProps.name === nextProps.name && prevProps.count === nextProps.count; } );

When to use:

  • Components that render frequently with the same props
  • Components with expensive render logic
  • Components that receive primitive props (strings, numbers, booleans)

When NOT to use:

  • Props change frequently (memo overhead may outweigh benefits)
  • Props are complex objects/arrays (shallow comparison may not work as expected)
  • Component is already fast and simple

2. useMemo - Memoize Expensive Calculations

useMemo memoizes the result of a computation and only recalculates when dependencies change.

function ExpensiveList({ items, filter }) { const filteredItems = useMemo(() => { console.log('Filtering items...'); return items.filter(item => item.category === filter); }, [items, filter]); const sortedItems = useMemo(() => { console.log('Sorting items...'); return [...filteredItems].sort((a, b) => a.price - b.price); }, [filteredItems]); return ( <ul> {sortedItems.map(item => <li key={item.id}>{item.name}</li>)} </ul> ); } function FibonacciComponent({ n }) { const fibonacci = useMemo(() => { console.log('Calculating fibonacci...'); if (n <= 1) return n; let a = 0, b = 1; for (let i = 2; i <= n; i++) { [a, b] = [b, a + b]; } return b; }, [n]); return <div>Fibonacci({n}) = {fibonacci}</div>; }

When to use:

  • Expensive calculations (filtering, sorting, mathematical operations)
  • Derived data that depends on props/state
  • Creating objects/arrays that are passed as props to memoized components

When NOT to use:

  • Simple calculations (overhead may be greater than benefit)
  • Dependencies change on every render (defeats the purpose)
  • Primitive values (no need to memoize)

3. useCallback - Memoize Callback Functions

useCallback returns a memoized version of a callback that only changes if dependencies change. This prevents child components from re-rendering unnecessarily.

function ParentComponent({ userId }) { const [count, setCount] = useState(0); const [name, setName] = useState(''); const handleClick = useCallback(() => { console.log('Button clicked'); setCount(c => c + 1); }, []); const handleSubmit = useCallback((data) => { console.log('Submitting:', data); fetch(`/api/users/${userId}`, { method: 'POST', body: JSON.stringify(data) }); }, [userId]); const handleChange = useCallback((e) => { setName(e.target.value); }, []); return ( <div> <ExpensiveChildComponent onClick={handleClick} onSubmit={handleSubmit} /> <input value={name} onChange={handleChange} /> </div> ); } const ExpensiveChildComponent = React.memo(({ onClick, onSubmit }) => { console.log('Rendering ExpensiveChildComponent'); return ( <div> <button onClick={onClick}>Click me</button> <form onSubmit={onSubmit}>...</form> </div> ); });

When to use:

  • Callbacks passed to memoized child components (React.memo)
  • Callbacks used as dependencies in other hooks (useEffect, useMemo)
  • Callbacks passed to expensive child components

When NOT to use:

  • Callbacks that change on every render anyway (no benefit)
  • Simple callbacks that don't cause re-renders
  • Callbacks not passed to memoized components

Common Pitfalls and Best Practices:

function BadExample({ items }) { const [filter, setFilter] = useState(''); const filteredItems = items.filter(item => item.name.includes(filter)); const handleClick = () => { console.log('clicked'); }; return ( <MemoizedChild items={filteredItems} onClick={handleClick} /> ); } function GoodExample({ items }) { const [filter, setFilter] = useState(''); const filteredItems = useMemo(() => { return items.filter(item => item.name.includes(filter)); }, [items, filter]); const handleClick = useCallback(() => { console.log('clicked'); }, []); return ( <MemoizedChild items={filteredItems} onClick={handleClick} /> ); }

Other Optimization Techniques:

  • Code splitting: Lazy loading components with React.lazy() and Suspense to reduce initial bundle size
  • Virtualization: Use libraries like react-window or react-virtualized for long lists (only render visible items)
  • Avoid inline object/function creation in render: Creates new references each render, breaking memoization
  • React DevTools Profiler: Identify performance bottlenecks and measure optimization impact
  • Key optimization: Use stable keys in lists (avoid index as key when items can be reordered)
  • State lifting: Lift state up only when necessary, keep state as local as possible
  • Context optimization: Split contexts to prevent unnecessary re-renders

Performance Measurement:

import { Profiler } from 'react'; function onRenderCallback(id, phase, actualDuration) { console.log('Component:', id); console.log('Phase:', phase); console.log('Actual duration:', actualDuration); } function App() { return ( <Profiler id="App" onRender={onRenderCallback}> <YourComponent /> </Profiler> ); }
Q5: What is the Virtual DOM and how does React use it?

The Virtual DOM is a JavaScript representation of the real DOM. React uses it to:

  • Create a virtual representation of UI in memory
  • Compare new virtual DOM with previous version (diffing)
  • Calculate minimal changes needed (reconciliation)
  • Update only changed parts in the real DOM (batching)

This approach is more efficient than directly manipulating the DOM because it batches updates and minimizes expensive DOM operations.

Q6: Explain React's reconciliation algorithm.

Reconciliation is React's process of updating the DOM. Key points:

  • React compares new element tree with previous one
  • Uses "diffing" algorithm to find differences
  • Updates only changed nodes, not entire tree
  • Uses keys to identify which items changed, were added, or removed
  • Keys should be stable, unique, and predictable

Without keys, React may unnecessarily recreate components, causing performance issues and losing component state.

Q7: What are React Portals and when would you use them?

Portals allow you to render children into a DOM node outside the parent component hierarchy.

import { createPortal } from 'react-dom'; function Modal({ children }) { return createPortal( children, document.getElementById('modal-root') ); }

Use cases: modals, tooltips, dropdowns, or any UI that needs to break out of parent container constraints (e.g., overflow: hidden).

TypeScript

Q8: What are the main benefits of TypeScript over JavaScript?
  • Type safety: Catches errors at compile time
  • Better IDE support: Autocomplete, refactoring, navigation
  • Self-documenting code: Types serve as documentation
  • Refactoring confidence: Easier to refactor large codebases
  • Better tooling: Enhanced debugging and development experience
Q9: Explain TypeScript interfaces vs types.

Interfaces: Can be extended and merged, better for object shapes

interface User { name: string; age: number; } interface Admin extends User { role: string; }

Types: More flexible, can represent unions, intersections, primitives, and computed types

type Status = 'active' | 'inactive'; type UserOrAdmin = User | Admin;

Generally: use interfaces for object shapes, types for unions/intersections/primitives.

Q10: What are TypeScript generics and provide an example.

Generics allow you to create reusable components that work with multiple types.

function identity<T>(arg: T): T { return arg; } interface ApiResponse<T> { data: T; status: number; } const userResponse: ApiResponse<User> = { data: { name: 'John', age: 30 }, status: 200 };

Generics provide type safety while maintaining flexibility.

Q11: Explain TypeScript utility types (Partial, Pick, Omit, etc.).

Utility types transform existing types:

  • Partial<T>: Makes all properties optional
  • Pick<T, K>: Selects specific properties
  • Omit<T, K>: Removes specific properties
  • Required<T>: Makes all properties required
  • Readonly<T>: Makes all properties readonly
interface User { id: number; name: string; email: string; } type UserUpdate = Partial<User>; // All optional type UserPreview = Pick<User, 'id' | 'name'>; // Only id and name type UserWithoutId = Omit<User, 'id'>; // Everything except id

Redux & State Management

Q12: Explain Redux architecture and its core concepts.

Redux is a predictable state container. Core concepts:

  • Store: Single source of truth holding application state
  • Actions: Plain objects describing what happened
  • Reducers: Pure functions that specify how state changes in response to actions
  • Dispatch: Method to send actions to the store
  • Selectors: Functions to extract specific data from state

Data flow: Action → Dispatch → Reducer → New State → Components re-render

Q13: What is Redux Toolkit and why is it preferred over plain Redux?

Redux Toolkit (RTK) is the official recommended way to write Redux logic. Benefits:

  • configureStore: Simplified store setup with good defaults
  • createSlice: Reduces boilerplate for actions and reducers
  • createAsyncThunk: Handles async actions easily
  • Immer integration: Allows "mutating" logic in reducers
  • Less boilerplate: Significantly less code than plain Redux
import { createSlice } from '@reduxjs/toolkit'; const counterSlice = createSlice({ name: 'counter', initialState: { value: 0 }, reducers: { increment: (state) => { state.value += 1; // Immer allows direct mutation } } });
Q14: How do you handle async actions in Redux Toolkit?

Use createAsyncThunk for async operations:

import { createAsyncThunk, createSlice } from '@reduxjs/toolkit'; export const fetchUser = createAsyncThunk( 'user/fetch', async (userId) => { const response = await fetch(`/api/users/${userId}`); return response.json(); } ); const userSlice = createSlice({ name: 'user', initialState: { data: null, loading: false, error: null }, reducers: {}, extraReducers: (builder) => { builder .addCase(fetchUser.pending, (state) => { state.loading = true; }) .addCase(fetchUser.fulfilled, (state, action) => { state.loading = false; state.data = action.payload; }) .addCase(fetchUser.rejected, (state, action) => { state.loading = false; state.error = action.error.message; }); } });
Q15: When would you use Redux vs local component state vs Context API?
  • Local state (useState): Component-specific data, form inputs, UI state (modals, toggles)
  • Context API: Simple global state, theme, user auth, avoiding prop drilling
  • Redux: Complex application state, shared data across many components, time-travel debugging, middleware needs (logging, async), large applications

Rule of thumb: Start with local state, use Context for simple global state, use Redux for complex state management needs.

JavaScript (ES6+)

Q16: Explain closures in JavaScript.

A closure is a function that has access to variables in its outer (enclosing) scope even after the outer function has returned.

function outerFunction(x) { return function innerFunction(y) { return x + y; // innerFunction has access to x }; } const addFive = outerFunction(5); console.log(addFive(3)); // 8

Common use cases: data privacy, function factories, event handlers, callbacks.

Q17: What is the difference between var, let, and const?
  • var: Function-scoped, hoisted, can be redeclared
  • let: Block-scoped, hoisted but in TDZ (Temporal Dead Zone), cannot be redeclared, can be reassigned
  • const: Block-scoped, hoisted but in TDZ, cannot be redeclared or reassigned (but object properties can be modified)

Best practice: Use const by default, let when reassignment is needed, avoid var.

Q18: Explain promises and async/await.

Promises: Represent eventual completion of async operation

fetch('/api/data') .then(response => response.json()) .then(data => console.log(data)) .catch(error => console.error(error));

Async/await: Syntactic sugar for promises, makes async code look synchronous

async function fetchData() { try { const response = await fetch('/api/data'); const data = await response.json(); console.log(data); } catch (error) { console.error(error); } }
Q19: What is the event loop and how does it work?

The event loop enables JavaScript's asynchronous behavior:

  • JavaScript is single-threaded but uses an event loop for concurrency
  • Call stack executes synchronous code
  • Web APIs handle async operations (setTimeout, fetch, etc.)
  • Callback queue holds callbacks from completed async operations
  • Event loop moves callbacks from queue to stack when stack is empty

This allows non-blocking I/O operations despite being single-threaded.

Q20: Explain destructuring and spread operator.

Destructuring: Extract values from arrays/objects

const { name, age } = user; const [first, second] = array; const { name: userName, ...rest } = user;

Spread operator: Expand arrays/objects

const newArray = [...oldArray, newItem]; const newObject = { ...oldObject, newProp: value };

WebSockets & Real-time Communication

Q21: What are WebSockets and how do they differ from HTTP?

HTTP: Request-response protocol, client initiates, stateless, one-way communication

WebSockets: Full-duplex communication, persistent connection, server can push data, lower latency

const socket = new WebSocket('ws://example.com'); socket.onopen = () => { socket.send('Hello Server'); }; socket.onmessage = (event) => { console.log('Message:', event.data); }; socket.onerror = (error) => { console.error('Error:', error); }; socket.onclose = () => { console.log('Connection closed'); };

Use cases: Chat applications, real-time notifications, live updates, collaborative editing.

Q22: How would you handle WebSocket reconnection logic?
class WebSocketManager { constructor(url) { this.url = url; this.socket = null; this.reconnectAttempts = 0; this.maxReconnectAttempts = 5; this.reconnectDelay = 1000; } connect() { this.socket = new WebSocket(this.url); this.socket.onopen = () => { this.reconnectAttempts = 0; }; this.socket.onclose = () => { if (this.reconnectAttempts < this.maxReconnectAttempts) { setTimeout(() => { this.reconnectAttempts++; this.connect(); }, this.reconnectDelay * this.reconnectAttempts); } }; } }

Implement exponential backoff, connection state management, and message queuing for offline scenarios.

Testing (Playwright)

Q23: What is Playwright and how does it differ from other testing tools?

Playwright is an end-to-end testing framework. Advantages:

  • Cross-browser testing (Chromium, Firefox, WebKit)
  • Auto-waiting for elements (no manual waits needed)
  • Network interception and mocking
  • Mobile device emulation
  • Parallel test execution
  • Better debugging with trace viewer
  • Modern async/await API
import { test, expect } from '@playwright/test'; test('user login', async ({ page }) => { await page.goto('https://example.com'); await page.fill('#username', 'user'); await page.fill('#password', 'pass'); await page.click('button[type="submit"]'); await expect(page.locator('.welcome')).toBeVisible(); });
Q24: How do you handle async operations and waiting in Playwright?

Playwright has built-in auto-waiting, but you can also use explicit waits:

await page.waitForSelector('.element'); await page.waitForLoadState('networkidle'); await page.waitForResponse(response => response.url().includes('/api/data')); await expect(page.locator('.element')).toBeVisible({ timeout: 5000 });

Playwright automatically waits for elements to be actionable before interacting with them.

Webpack & Build Tools

Q25: What is Webpack and what problem does it solve?

Webpack is a module bundler that:

  • Bundles JavaScript modules and dependencies
  • Transforms code (Babel, TypeScript, SASS)
  • Optimizes assets (minification, tree-shaking, code splitting)
  • Manages assets (images, fonts, CSS)
  • Enables hot module replacement (HMR) for development

It takes modules with dependencies and generates static assets representing those modules.

Code splitting allows you to split code into smaller chunks loaded on demand:

  • Entry point splitting: Multiple entry points
  • Dynamic imports: Using import() for lazy loading
  • SplitChunksPlugin: Automatically splits common dependencies
// Dynamic import const LazyComponent = React.lazy(() => import('./LazyComponent')); // webpack.config.js module.exports = { optimization: { splitChunks: { chunks: 'all', cacheGroups: { vendor: { test: /[\\/]node_modules[\\/]/, name: 'vendors', chunks: 'all', }, }, }, }, };

API Integration & HTTP

Q27: How do you handle API requests with Axios interceptors?
import axios from 'axios'; const api = axios.create({ baseURL: 'https://api.example.com', }); api.interceptors.request.use( (config) => { const token = localStorage.getItem('token'); if (token) { config.headers.Authorization = `Bearer ${token}`; } return config; }, (error) => Promise.reject(error) ); api.interceptors.response.use( (response) => response, (error) => { if (error.response?.status === 401) { localStorage.removeItem('token'); window.location.href = '/login'; } return Promise.reject(error); } );

Interceptors are useful for: adding auth tokens, error handling, request/response transformation, logging.

Q28: Explain REST API principles.

REST (Representational State Transfer) principles:

  • Stateless: Each request contains all information needed
  • Resource-based: URLs represent resources
  • HTTP methods: GET (read), POST (create), PUT (update), DELETE (remove)
  • Status codes: 200 (success), 201 (created), 400 (bad request), 401 (unauthorized), 404 (not found), 500 (server error)
  • JSON/XML: Standard data formats

Q34: What is tree shaking and how does it work?

Tree shaking eliminates dead code (unused exports) from the final bundle:

  • Requires ES6 modules (import/export)
  • Static analysis determines unused code
  • Webpack and modern bundlers do this automatically
  • Reduces bundle size significantly

Example: If you import only one function from a library, only that function is included in the bundle.

General Frontend Concepts

Q35: Explain CORS and how to handle it.

CORS (Cross-Origin Resource Sharing) allows servers to specify who can access resources:

  • Same-origin: Same protocol, domain, port
  • Cross-origin: Different origin requires CORS headers
  • Preflight: OPTIONS request for complex requests

Server must send: Access-Control-Allow-Origin, Access-Control-Allow-Methods, Access-Control-Allow-Headers

Client-side: Use proxy in development, configure server properly in production.

Q36: What is the difference between authentication and authorization?
  • Authentication: Verifying who you are (login, JWT tokens, sessions)
  • Authorization: Verifying what you can do (permissions, roles, access control)

Example: Authentication confirms you're logged in, authorization determines if you can delete a post.

Q37: How do you handle errors in a React application?
  • Error Boundaries: Catch errors in component tree
  • Try-catch: For async code and event handlers
  • Error logging: Services like Sentry for production
  • User-friendly messages: Display helpful error messages
  • Fallback UI: Show alternative content on error
class ErrorBoundary extends React.Component { state = { hasError: false }; static getDerivedStateFromError(error) { return { hasError: true }; } componentDidCatch(error, errorInfo) { console.error('Error:', error, errorInfo); } render() { if (this.state.hasError) { return <h1>Something went wrong.</h1>; } return this.props.children; } }
Q38: Explain responsive design principles.
  • Mobile-first: Design for mobile, then scale up
  • Flexible layouts: Use relative units (%, em, rem, vw, vh)
  • Media queries: Adapt styles for different screen sizes
  • Flexible images: max-width: 100%, object-fit
  • CSS Grid & Flexbox: Modern layout techniques
  • Viewport meta tag: Proper scaling on mobile devices
Q41: Explain Cookies, Local Storage, and Session Storage. What are they used for and what are the differences?

Cookies:

  • What it is: Small pieces of data stored by the browser, sent with every HTTP request to the server
  • Storage size: ~4KB per cookie, limited number per domain
  • Lifespan: Can be set with expiration date (persistent) or session-based (deleted when browser closes)
  • Access: Accessible on both client and server (sent automatically with requests)
  • Use cases: Authentication tokens, session management, tracking, user preferences, shopping cart data
  • Security: Can be secured with HttpOnly, Secure, SameSite flags
document.cookie = "username=John; expires=Thu, 18 Dec 2025 12:00:00 UTC; path=/"; document.cookie = "token=abc123; Secure; HttpOnly; SameSite=Strict"; function getCookie(name) { const value = `; ${document.cookie}`; const parts = value.split(`; ${name}=`); if (parts.length === 2) return parts.pop().split(';').shift(); }

Local Storage:

  • What it is: Web Storage API that stores data locally in the browser with no expiration
  • Storage size: ~5-10MB per origin (varies by browser)
  • Lifespan: Persists until explicitly cleared by user or programmatically
  • Access: Only accessible via JavaScript on the client-side (not sent to server automatically)
  • Use cases: User preferences, cached data, offline data, application state, theme settings
  • Scope: Per origin (protocol + domain + port)
  • API: Synchronous API (blocks main thread)
localStorage.setItem('user', JSON.stringify({ name: 'John', id: 123 })); const user = JSON.parse(localStorage.getItem('user')); localStorage.removeItem('user'); localStorage.clear(); localStorage.setItem('theme', 'dark'); const theme = localStorage.getItem('theme');

Session Storage:

  • What it is: Web Storage API similar to Local Storage but scoped to browser tab/window session
  • Storage size: ~5-10MB per origin (varies by browser)
  • Lifespan: Persists only for the duration of the tab/window session (deleted when tab closes)
  • Access: Only accessible via JavaScript on the client-side, not shared between tabs
  • Use cases: Temporary form data, multi-step form state, temporary UI state, tab-specific data
  • Scope: Per origin AND per tab/window (each tab has its own session storage)
  • API: Same API as Local Storage (synchronous)
sessionStorage.setItem('formData', JSON.stringify({ step: 2, data: {...} })); const formData = JSON.parse(sessionStorage.getItem('formData')); sessionStorage.removeItem('formData'); sessionStorage.clear();

Key Differences:

Feature Cookies Local Storage Session Storage
Size Limit ~4KB ~5-10MB ~5-10MB
Lifespan Set expiration or session Until cleared Tab/window session
Server Access Yes (sent with requests) No No
Scope Per domain Per origin Per origin + tab
API document.cookie localStorage.* sessionStorage.*

Best Practices:

  • Cookies: Use for authentication tokens, session IDs, small data that server needs. Set HttpOnly for sensitive data to prevent XSS attacks
  • Local Storage: Use for user preferences, cached data, offline functionality. Never store sensitive data (passwords, tokens) without encryption
  • Session Storage: Use for temporary data, form state, tab-specific data. Perfect for multi-step forms or temporary UI state
  • Security: Always validate and sanitize data. Use HTTPS. Be aware of XSS vulnerabilities. Consider using httpOnly cookies for sensitive tokens
  • Performance: Local/Session Storage operations are synchronous and block the main thread. For large data, consider IndexedDB

When to use each:

  • Cookies: Authentication, server-side session management, tracking (with consent), small preferences
  • Local Storage: User settings, theme preferences, cached API responses, offline data, application state persistence
  • Session Storage: Form drafts, temporary UI state, tab-specific data, multi-step processes

Architecture & Best Practices

Q39: How do you structure a large React application?

Common patterns:

  • Feature-based: Group by features (auth/, dashboard/, profile/)
  • Component organization: Components, hooks, utils, types
  • Shared components: Common UI components in shared/
  • State management: Redux slices or Context providers
  • API layer: Centralized API calls and types
  • Constants: Configuration and constants
  • Tests: Co-located or separate test directory
Q40: What are some React best practices you follow?
  • Use functional components and hooks
  • Keep components small and focused (single responsibility)
  • Extract reusable logic into custom hooks
  • Use TypeScript for type safety
  • Implement proper error boundaries
  • Optimize re-renders with memoization
  • Use meaningful component and variable names
  • Follow consistent code formatting (Prettier, ESLint)
  • Write tests for critical functionality
  • Document complex logic
  • Use proper key props in lists
  • Avoid prop drilling (use Context or state management)
